## Phase 1: Rendering stability (v802–v807)

### v802 — Portview collapse observer lifecycle

**Problems:** pace bar missing on PWA, KAM name wraps 2 rows, SKU movement slow (admin), portview collapses after filter/search, Echo→portview jank.

**Root causes found:**
- `_pvCollapseObserver` (scroll-collapse rAF loop) not disconnected on nav-away → 2+ observers running → rAF storm
- Admin/TL bundle fetch using admin email as key → 404 → 90s timeout (SKU movement delay)
- scroll-clamp: filter shrinks list → browser clamps scrollY=0 → frame() reads "still collapsed" → stuck

**Fixes (src/06_portview_teamview.js, src/05_kam_view.js, src/styles_portview.css):**
- Generation token `_pvObsGen` kills overlapping observers
- `showScreen()` teardown on nav-away, clears `_pvLastCollapseMs`/`_pvWasCollapsed`
- not-scrollable guard always re-expands after list shrinks
- `setPvSort()`/`setPvView()` snapshot collapse state (parity with search path)
- Admin/TL bundle retry resolves correct KAM email (not admin email)
- `#portview-heading` truncates to 1 line with ellipsis
- 41 unit tests incl. clamp-aware flow sims — all passing

### v803 — Safari iOS crash: invalid setAttribute

**Problem:** `renderPortviewTargetBar()` hero card empty box on every iOS session, every role.

**Console diagnostic:** `[Error] InvalidCharacterError: Invalid qualified name: '-webkit-tap-highlight-color'`

**Root cause:** `bar.setAttribute('-webkit-tap-highlight-color', 'transparent')` — attribute name starting with `-` is invalid per DOM spec. Safari strict, Chrome lenient. Async function aborts before reaching `bar.innerHTML = ...`.

**Fix (src/07b_nrr_target.js):** `bar.style.webkitTapHighlightColor = 'transparent'`

### v804 — hero card value guard: empty box when target allocated

**Root cause:** `bar.className = 'fb-team'` + `bar.style.display = 'block'` were set **before** value guard check. If guard returned early (same renderKey), bar showed as visible CSS box with no innerHTML.

**Fix:** moved `bar.className` + `display:block` + `opacity` to **after** value guard. If guard fires (content unchanged), still sync class+visibility.

### v805 — Save page SKU detail sheet under pb-footer

**Root cause:** iOS Safari creates new stacking context on `position:fixed` + `transform` with X translation (`translateX`). `pb-footer` (z-index:99) rendered above `data-panel` (z-index:201) due to stacking context isolation.

**Fix (src/styles_save.css, src/styles_layout.css):** Replace `left:50% + translateX(-50%)` centering with `left:0 + right:0 + margin:0 auto`. `translateY` only for show/hide animation.

New z-index stack: `bnav(170) → pb-footer(210) → data-overlay(220) → data-panel(221) → verify-sheet(301)`

### v806 — detail-sheet hidden under bnav

**Console diagnostic confirmed:** `detail-sheet` z-index:161 < `bnav` z-index:170.

**Fix (src/styles_portview.css):** `detail-overlay: 160→175`, `detail-sheet: 161→176`

### v807 — Skills tab disappears after Save → portview

**Root cause:** `body.kam-sense-active` added on Save entry, removed only when `_kamSenseReturn === true`. If user entered Save without account selected, `_kamSenseReturn` never set → class stuck → `NavConfig hideWhen:['kam-sense-active']` hides Skills.

**Fix (src/12_nav_config.js):** `showScreen()` wrapper calls `renderNav()` when navigating back to portview/overview/teamview/skills. Runs after `_origShow` (which cleans body classes) so renderNav sees final state.

### v810 — Fixed toolbar row + iOS segmented control + concept D expand button

**Toolbar row (qnrr-toolbar):** single fixed row between hero and chart-zone, consistent position in both views.

**Left: iOS segmented control (`qnrr-vseg`):**
- Single pill, thumb slides L↔R via CSS transform
- Icon-only (bar-chart icon | list icon), no text labels

**Note:** `qnrr-vseg` naming chosen specifically to avoid class collision with bar chart segment divs (which use `qnrr-seg`).

### v811 — Stacked bar segments tight + neutral adj-note

- Removed `.qnrr-seg + .qnrr-seg { border-top: 1px solid }` — inter-segment gap
- Removed `border-top` from `.qnrr-base-hov-seg` (handover layer in base bar)
- `qnrr-bar-body` wrapper border-radius:5px preserved → top corners still rounded
- `qnrr-base-adj-note` (ฐานปรับจาก...): red → neutral blue-muted

### v812 — Fix stacked bar: CSS class name collision (root cause)

**Root cause (git blame → db6929c359 v788):** Both the iOS toolbar pill (`class="qnrr-seg"`) and bar chart segment divs (`class="qnrr-seg"`) used the same class name. Toolbar CSS defined `border-radius:8px; display:flex; position:relative` which bar segments inherited → each segment got its own rounded corners → "stacked separately / แปะๆ" look.

Removing `border-top` in v811 was correct but didn't fully fix it — `border-radius` was still applying from the collision.

**Fix:** Renamed toolbar class `qnrr-seg → qnrr-vseg` across `shell.html` and `styles_qnrr.css`. IDs unchanged (`qnrr-seg`, `qnrr-seg-chart` etc.) so `getElementById` calls in JS unaffected. Bar segments keep `class="qnrr-seg"` with only `width:100%; min-height:3px`.
