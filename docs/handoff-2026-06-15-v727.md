### 1. Echo v2 Architecture (Worker v4)

**Worker: `worker/freshket-sense-ai-proxy.js`**

เปลี่ยนจาก v3 (Gemini audio native) เป็น 3-step pipeline:

| Endpoint | ทำอะไร | Model |
|---|---|---|
| `/transcript` | audio → segments[] + segment_id + speaker_confidence + transcript_confidence | Groq Whisper → Gemini Flash Lite diarize |
| `/summarize` | segments[] → notes + customer_said + next_steps + tone | Gemini 2.5 Flash Lite |
| `/analyze` | segments[] + summary → 14 skills + OCPB + evidence with segment_id | Claude Sonnet |
| `/eval` | วัด criteria ตาม spec (Thai accuracy, hallucination, evidence coverage) | Claude Haiku |

**Fallback chain (spec principle: transcript never lost):**
- Groq Whisper fail → retry 3 ครั้ง
- Gemini diarize fail → fallback ใช้ Whisper raw segments, speaker = "ไม่ทราบ"
- `/analyze` retry รวม JSON parse fail

**Model ที่ใช้:**
- Groq: `whisper-large-v3`
- Gemini: `gemini-2.5-flash-lite` (เปลี่ยนจาก 2.0-flash-lite ที่ shutdown 2026-06-01)
- Claude: `claude-sonnet-4-6`

**Groq prompt (anti-hallucination):**
```
บทสนทนาภาษาไทย ระหว่างพนักงานขายกับเจ้าของร้านอาหาร
```
สั้นและไม่มีคำ brand name — ป้องกัน Whisper hallucinate prompt กลับมาใน transcript

### 2. `09_conv_intel.js` — Pipeline Rewrite (v717-v727)

**Save architecture ใหม่ (Echo v2):**
```
Step 1: transcript done → _saveTranscriptOnly() → insert ci_sessions (stage='transcribed')
Step 2: summary done → update _lastResult
Step 3: analysis done → _saveAnalysisToExistingSession() → update row เดิมด้วย skills+OCPB
```

**สิ่งที่แก้:**
- `_processBlob` rewrite — save transcript ก่อน analysis เสมอ
- `_callTranscript` dynamic timeout: `max(6min, recording_secs × 1.5)`
- Summary fail = null (non-fatal), analysis ยังเดินต่อ
- `_lastResult` null guard ก่อน update panels (ป้องกัน crash เมื่อ user ปิด sheet)
- Recovery feature: `_checkRecoverBuffer()` แสดง banner "พบการบันทึกค้าง" + ปุ่ม "วิเคราะห์ต่อ"
- `_sessionId` guard ใน `open()` — ไม่ reset ถ้า pipeline กำลัง save

### 5. Issues ที่ยังค้างอยู่ (สำคัญ)

#### ❌ Save pipeline ยังไม่ทำงานถูกต้อง (v727)

**Symptom:** pipeline เดินครบ 3 steps แต่ `_sessionId` เป็น null ตอน save step → fallback insert สร้าง row ใหม่แทน update row transcript เดิม

**Root cause ที่วิเคราะห์ได้:**
`open()` ถูกเรียกซ้ำระหว่าง pipeline ทำงาน → reset `_sessionId = null` → save เป็น orphan row

**สาเหตุที่ `open()` ถูกเรียกซ้ำ:**
`_checkRecoverBuffer()` ที่เพิ่มใน v723 เรียกผ่าน `setTimeout(_checkRecoverBuffer, 300)` ใน `open()` → เจอ IDB chunks → trigger recovery flow → เรียก `open()` อีกครั้ง

**วิธีแก้ที่ถูก (ยังไม่ได้ implement):**
1. ย้าย `setTimeout(_checkRecoverBuffer, 300)` ออกจาก `open()` ไปใส่ใน `_mount()` แทน
2. ใน `_checkRecoverBuffer()`: เพิ่ม `if (_phase !== 'idle' || _sessionId) return;` — ไม่ recover ถ้า session กำลังทำงานอยู่

**Code location:** `src/09_conv_intel.js`
- `_checkRecoverBuffer` definition: ~line 47570
- `setTimeout(_checkRecoverBuffer, 300)` call: ใน `open()` function ท้ายสุด
