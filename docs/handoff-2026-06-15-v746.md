## Sales Boot Fix — Root Cause & Fix Chain

### Root Cause ที่แท้จริง (พบที่ v745)

ปัญหาเกิดจาก **3 จุดที่ซ้อนกัน** ทั้งหมดเกิดจาก timing race ระหว่าง role detection กับ R2 fetch:

**1. `R2_FILES` เป็น frozen object (`Object.freeze` ใน shell.html)**
`_patchR2FilesForSales()` พยายาม mutate frozen object → silent no-op ทุกครั้ง → Sales โหลด KAM CSV

**2. `DataRegistry._checkTier(1)` ใช้ `_isSales()` ที่เช็ค `body.sales-mode`**
class นี้ถูก set ใน `_splashPreFade` (หลัง splash fade) ไม่ใช่ตอน boot → Tier 1 ไม่ ready ตลอด session

**3. `DataRegistry._isSales()` เป็น dead code path ที่ unreliable**
body.sales-mode ไม่ set → include handover ใน Tier 1 → handover 404 → `_tierReady[1]` ไม่เคย set → `_flush HELD` ตลอด

### Fix ที่ทำ

| Version | Fix |
|---|---|
| v738 | `_salesR2Override` map แทน frozen `R2_FILES` mutation + `_getR2File()` |
| v741 | `allCriticalReady()` กลับ v643 pattern — check `_cloudLoadedTabs` โดยตรง |
| v743 | bg ETag skip handover สำหรับ Sales |
| v744 | `_detectAndMarkSalesSession()` — patch Sales R2 ทันทีหลังทุก profile set |
| v745 | `DataRegistry._checkTier(1)` delegate ไป `allCriticalReady()` แทน `_isSales()` |
| v746 | Clean up dead code + redundant code ทั้งหมด |

### Architecture หลัง v746

```
Role detection flow:
  profile set → normalizeCurrentUserProfileRole() → _detectAndMarkSalesSession()
    → _patchR2FilesForSales() → _salesR2Override populated ทันที

Sales signal hierarchy (single source):
  _salesR2Override.length > 0  ← authoritative (set ก่อน fetch ทุกกรณี)
  getCurrentRole() === 'sales'  ← fallback

Data gate:
  allCriticalReady() ← Sales = portview+history, KAM = portview+history+handover
  DataRegistry._checkTier(1) ← delegates to allCriticalReady() (v745)
  RenderBus._flush ← uses allCriticalReady() (v741)
```
