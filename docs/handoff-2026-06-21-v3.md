## Architecture

**Scope:** KAM / PM / ADMIN เท่านั้น SALE นับแยกจาก dwh.order โดยตรง B2C ออกทั้งหมด

**Grain:** 1 row ต่อ outlet ต่อ period_month ต่อ portfolio
- + transfer_out row (curr_gmv=0) สำหรับ portfolio ต้นทาง
- + intra transfer_out / transfer_in rows สำหรับกรณี staff เปลี่ยนภายใน portfolio เดียวกัน

**Mar Cohort (fixed denominator):**
- commercial_owner IN ('KAM','PM','ADMIN') + gmv_ex_vat > 0 ใน Mar
- filter เฉพาะ KAM/PM/ADMIN orders (ignore SALE orders ใน Mar)
- exclude outlet ที่ new_user_exp_date = Mar (handover ใน Mar)

**Transfer Layers:**
- `intra`: staff เปลี่ยนในพอร์ตเดิม (Dent→Pop ใน KAM) → generate 3 rows: core_nrr + transfer_out(intra) + transfer_in(intra)
- `inter`: ข้ามพอร์ต (KAM→PM) → transfer_in/out symmetric
- `external`: ออกไป SALE → transfer_out เท่านั้น ไม่มี pair

**Columns สำคัญ:**
- `movement_type`: core_nrr, core_nrr_churn, handover, new_sales, expansion, comeback, transfer_in, transfer_out
- `transfer_scope`: intra / inter / external / NULL
- `from_portfolio`, `to_portfolio`: มีค่าเฉพาะ transfer rows
- `from_staff_owner`, `to_staff_owner`: มีค่าเฉพาะ intra transfer rows
- `base_gmv`: Mar GMV ของ outlet (NRR denominator)
- `curr_gmv`: GMV จริงของ period

## Movement Definitions (validated จาก commission SQL + reconcile SQL)

| movement | definition |
|---|---|
| expansion | first_dollar >= Apr + commercial_owner ≠ SALE ณ period |
| handover | โอนจาก SALE ใน Mar (new_user_exp_date=Mar) หรือ fallback pre_apr=SALE + มี Mar GMV |
| new_sales | โอนจาก SALE ใน Q (new_user_exp_date Apr/May/Jun) |
| core_nrr | อยู่ mar_cohort + same portfolio + curr_gmv > 0 |
| core_nrr_churn | อยู่ mar_cohort + same portfolio + curr_gmv = 0 |
| comeback | ไม่มี Mar GMV + first_dollar < Apr + กลับมาซื้อใน Q — ไม่สน pre_period owner |
| transfer_in | รับโอนเข้า (inter หรือ intra) |
| transfer_out | คู่ตรงข้าม transfer_in เสมอ, curr_gmv = 0 |

**VP View (KAM+PM+ADMIN):**
- transfer inter = net zero (หักล้างกัน)
- transfer_out ที่มีความหมาย = external เท่านั้น (→ SALE)

## CTEs หลัก (ตามลำดับ)

```
params → outlet_first_dollar → base_gmv → apr/may/jun_gmv
→ mar/apr/may/jun_own → pre_mar/apr/may/jun_own
→ mar_handover_outlets (any Mar order ที่ new_user_exp=Mar)
→ mar_any_gmv (outlets ที่มี Mar GMV จาก owner ใดก็ได้)
→ mar_cohort
→ apr_labels (lock classification Apr)
→ may_labels (lock classification May — inherit จาก Apr ถ้า portfolio ไม่เปลี่ยน)
→ apr_rows / may_rows / jun_rows (LEG A + LEG A-INTRA + LEG B)
→ all_rows → FINAL SELECT
```

**LEG A:** outlets ที่มี order ใน period (portfolio scope)
**LEG A-INTRA:** intra staff transfer rows (detect จาก base_staff_owner ≠ current_staff_owner)
**LEG B:** mar_cohort ที่ไม่มี order ใน portfolio เดิม → core_nrr_churn หรือ transfer_out

## Priority Classification [1]→[7]

ใช้ทุก block (apr_labels, may_labels fallback, jun_rows fallback):

```
[1] expansion    : first_dollar >= Apr + owner ≠ SALE
[2] handover     : mho IS NOT NULL + port match Mar last owner
                   OR new_user_exp = Mar
[2b] handover   : no exp_date + pre_period = SALE
[2c] handover   : pre_period = SALE + มี Mar GMV (mar_any_gmv) + exp ไม่ใช่ Q
[3] new_sales    : new_user_exp ใน Q + pre_mar = SALE/NULL
[4] core         : mar_cohort + same portfolio
[5] transfer_in  : mar_cohort + diff portfolio (inter)
[6] comeback     : mc IS NULL + first_dollar < Apr + exp ไม่ใช่ Q (ไม่สน pre_period owner)
[7] transfer_in  : ELSE
```

**Carry-forward (May/Jun):**
- inherit fixed_label จาก Apr ถ้า `current_portfolio ยังเดิม`
- ถ้า portfolio เปลี่ยน → re-classify ใหม่ [1]→[7]

## Key SQL Principles

```sql
-- Join key dwh.order → dim.user_master
CAST(o.user_id AS STRING) = um.res_id

-- commercial_owner มาจาก dwh.order เสมอ ห้ามใช้ dim.user_master (current snapshot)
-- staff_owner มาจาก dwh.order (historical signal)

-- Day-1 lag
DATE_SUB(CURRENT_DATE('Asia/Bangkok'), INTERVAL 1 DAY)

-- Deduplication per outlet per month
QUALIFY ROW_NUMBER() OVER (PARTITION BY o.user_id ORDER BY o.delivery_date DESC) = 1

-- No temp tables — ใช้ inline UNNEST CTEs แทน
```
